**The original 90 s @ 4 workers laptop target was relaxed** after the
first measurement landed (1.84× speedup, not the 2.5× the target
assumed). Reason: Amdahl-bounded by the still-serial index-build pass
which accounts for ≈30% of run time. Parallelizing the index-build is
deferred to a follow-up ADR. Production hardware (big-org servers,
≥ 16 cores, faster NVMe) should exceed the original target naturally —
the per-worker work scales near-linearly once the index is built.

## Methodology

- Hardware: maintainer's laptop, Apple Silicon, SSD, 32 GiB RAM.
- Python: pyenv 3.12.7.
- Fixture: `tests/fixtures/synth_003000000_seed42_a.dat` / `_b.dat`
  generated by `tests.synthetic_data.generate_pair(3_000_000, 42, ...)`.
  ~2.95M records per file at ~1.34 GiB each. Sorted input.
- Engine config: stock `config/` (blake2b 16-byte digest,
  `input_sorted = true`, key_range `[4, 16)` on TU4R, `parallel_workers: 8`).
- Wall time: `time.perf_counter()` around `pipeline.run` / `pipeline.run_parallel`.
- Peak RSS: `resource.getrusage(RUSAGE_SELF).ru_maxrss` (bytes on
  macOS, KiB on Linux — handled by the bench harness).
- Each cell is a single hot-cache run. Numbers are reproducible to
  about ±5%; for acceptance-grade reporting in a production
  environment, average ≥ 3 runs.

### Speedup analysis

The speedup plateaus because the index-build pass is still
single-threaded. Amdahl's law with serial fraction `s ≈ 0.30`
predicts:

- N=4: max = 1 / (0.30 + 0.70/4) = **2.10×** — we hit 1.84×.
- N=8: max = 1 / (0.30 + 0.70/8) = **2.58×** — we hit 2.13×.

So we're within ~85% of the architectural ceiling. Future work to
parallelize the index-build (and possibly use shared memory for the
offset dicts instead of pickling) would lift the ceiling and the
realized speedup together.

### Time-budget breakdown (estimated)

For the 228.5 s single-process run:

- Index-build pass: ~70 s (30%)
- Inner-join (seek + parse + compare + write): ~140 s (60%)
- Output write + summary: ~20 s (10%)

The inner-join is what `Track A` parallelizes. The 124.5 s at 4
workers ≈ 70 s (still-serial index) + 140/4 s (parallel join) +
~20 s (write + merge) = ~125 s, matching observation.

## Track B — field-level normalizer

Identical-output verification on the realistic 10/11-record fixture
(test: `tests/test_field_integration.py::test_field_config_classifies_records_same_as_position_config`).
Each side runs the engine end-to-end with two configs that encode the
same exclusion semantics — one position-based (the stock config),
one field-based (a per-test fixture config with CL01 layout
`prefix(11) + timestamp(8, exclude) + suffix(2) + padding(39)` and
ENDS layout `segment_count(3, exclude)`).

| Config       | matches | mismatches | only_a | only_b | dups_a | dups_b |
|--------------|--------:|-----------:|-------:|-------:|-------:|-------:|
| Position     | 4       | 3          | 1      | 2      | 2      | 2      |
| Field        | 4       | 3          | 1      | 2      | 2      | 2      |
| **Delta**    | 0       | 0          | 0      | 0      | 0      | 0      |

`*.dat` outputs are byte-identical between the two configs. The
canonical hash bytes differ (position hashes the byte slice; field
hashes `<name>=<value>\x1F<name>=<value>...`), but the equivalence
relation each config encodes is the same, so per-pair match/mismatch
decisions agree everywhere.
