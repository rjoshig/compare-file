## Methodology

- Hardware: maintainer's local machine (Apple Silicon, SSD, 32 GiB RAM).
- Python: pyenv 3.12.7.
- Fixture: `tests/fixtures/synth_003000000_seed42_a.dat` / `_b.dat`
  generated by `tests.synthetic_data.generate_pair(3_000_000, 42, ...)`.
  ~2.95M records per file at ~1.34 GiB each. Sorted input.
- Engine config: `config/` defaults (blake2b 16-byte digest,
  `input_sorted = true`, key_range `[4, 16)` on TU4R).
- Wall time: `time.perf_counter()` around `pipeline.run`.
- Peak RSS: `resource.getrusage(RUSAGE_SELF).ru_maxrss` (bytes on
  macOS, KiB on Linux — handled by the bench harness).
- Each cell is a single hot-cache run unless noted. Final acceptance
  measurements should average ≥ 3 runs (TODO once Track A lands).

### Time breakdown (estimated, not yet profiled)

Rough split based on operation cost analysis:

- ~30% index-build pass (streaming parse of both files)
- ~60% inner-join (seek + reparse for each of ~2.77M joined keys)
- ~10% output write (8 files concurrently)

If those estimates hold, Track A's worker pool attacks the 60% slice
hardest — perfectly parallel after partitioning. The index-build pass
also parallelizes but with more IPC overhead.
